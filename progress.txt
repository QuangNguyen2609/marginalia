PHASE 3: Multi-Format Support (PDF, MOBI, DOCX)
-------------------------------------------------
Files: reader3.py (major additions), pyproject.toml

- Format router: process_book() dispatches by file extension
  - .epub -> process_epub() (existing)
  - .pdf  -> process_pdf() (new)
  - .docx -> process_docx() (new)
  - .doc  -> process_docx() (error: suggests converting to .docx)
  - .mobi -> process_mobi() (new)

- process_pdf() using PyMuPDF (fitz)
  - page.get_text("html") for formatted content per page
  - page.get_text() for plain text (LLM context)
  - page.get_images() + doc.extract_image() for image extraction
  - doc.get_toc() -> _build_pdf_toc() converts [(level, title, page)] to TOCEntry tree
  - Metadata from doc.metadata dict

- process_docx() using python-docx
  - Iterates paragraphs, detects Heading 1/2/3 styles
  - Groups paragraphs between headings into chapters
  - Extracts images from relationship parts
  - Builds TOC from heading hierarchy
  - Fallback: single chapter if no headings found
  - .doc files: raises clear error suggesting conversion

- process_mobi() using mobi library
  - mobi.extract() to get HTML content from MOBI container
  - Parses extracted HTML with BeautifulSoup
  - Splits content at h1/h2/h3 headings into chapters
  - Cleans HTML with existing clean_html_content()
  - Temp directory cleanup in finally block

- New dependencies added to pyproject.toml:
  - PyMuPDF>=1.24.0
  - python-docx>=1.1.0
  - mobi>=0.3.3
  - httpx>=0.28.0
  - python-multipart>=0.0.9

- All formats produce the same Book/ChapterContent/TOCEntry/BookMetadata dataclasses

SESSION 2: UI Improvements
---------------------------
Files: templates/reader.html, templates/base.html, server.py

Fix 1 — Chat Toggle Button Jump (reader.html)
- Root cause: body.no-transition rule did not cover .chat-toggle-btn
- .chat-toggle-btn has transition: opacity 0.2s; when chat-open is in localStorage,
  setChat(true) on load fades the button out visibly before transitions are suppressed
- Fix: added .chat-toggle-btn to the body.no-transition selector block
- Also added .chapter-counter to the same rule (for the new counter element)

Fix 2 — Remove Opacity Slider (base.html)
- Removed glass-opacity localStorage read from the early <head> script
- Changed --glass-opacity default from 0.6 to 1 in :root CSS
- Removed <label> + <input type="range"> opacity slider from toolbar HTML
- Removed opacitySlider JS variable and event listener
- Glass panels always render at full opacity; backdrop blur still active

Fix 3 — Continuous Scroll: All Chapters on One Page (server.py, reader.html)
- server.py: both /read/{book_id} and /read/{book_id}/{chapter_index} routes now pass
  all_chapters=book.spine and initial_chapter=chapter_index to the template instead
  of current_chapter/prev_idx/next_idx
- reader.html CSS: removed .chapter-nav/.nav-btn/.chapter-info; added .chapter-section
  (padding/divider between chapters) and .chapter-counter (fixed bottom-center pill)
- reader.html HTML: replaced single chapter block with {% for chapter in all_chapters %}
  loop rendering <section id="chapter-N" data-chapter-index="N"> elements; removed
  Previous/Next nav div; added chapter counter element
- reader.html TOC macro: removed server-side is_active check; added data-href attribute
  for JS-driven active state via IntersectionObserver
- reader.html JS: findAndGo() now calls scrollToChapter() (instant scrollIntoView)
  instead of window.location.href; IntersectionObserver tracks which chapter is in the
  upper viewport zone and updates counter + TOC active link; progress key changed from
  per-chapter to per-book; chat sends currentChapterIndex; initial TOC active link set
  on load

Bug Fix — Nested <style> Tag CSS Parsing (reader.html)
- Root cause: reader.html's {% block extra_styles %} injects a <style> tag inside
  base.html's outer <style> tag. The HTML parser correctly closes the outer <style>
  at the first </style>, so both tags are parsed. However, the CSS parser sees the
  literal text "<style>" inside the CSS stream and uses error recovery: it treats
  "<style>" + the next rule selector ("body") as one invalid qualified rule prelude,
  consuming "body { display: flex; height: 100vh; overflow: hidden; }" as its block
  and discarding it. All subsequent rules parse correctly.
- Fix: added a sacrificial :root {} rule as the first rule after <style> so that
  error recovery consumes "<style> :root {}" as the invalid rule, and
  "body { display: flex; ... }" is then parsed correctly as its own rule.

Bug Fix — Flex Layout Not Constraining #main Height (reader.html)
- Root cause: with all chapters rendered, #main's content height grew to ~440k px.
  In a flex row container, flex items have min-height: auto by default, which means
  the minimum height equals the content height. This overrides align-items: stretch,
  so #main expanded to full content height instead of being constrained to 100vh.
  Consequently the window scrolled instead of #main, breaking IntersectionObserver
  (which had root: mainEl) and progress save/restore (mainEl.scrollTop was always 0).
- Fix: added min-height: 0 to #main, allowing flex stretch to constrain it to the
  viewport height and enabling proper internal scrolling.

Fix 4 — Chapter Counter Sidebar Awareness (reader.html)
- Counter used position: fixed with left: 50%; transform: translateX(-50%) which
  is viewport-center, not reading-area-center. Opening the TOC or chat sidebar
  caused the counter to sit off-center.
- Fix: changed counter to left: 0; right: 0; text-align: center with margin-left/right
  transitions matching body.sidebar-open and body.chat-open rules (mirroring #main),
  so the counter stays centered in the reading area as sidebars open/close.

Fix 5 — Instant TOC Navigation (reader.html)
- scrollIntoView({ behavior: 'smooth' }) caused long animated scrolls when jumping
  between distant chapters (e.g. chapter 1 to chapter 20), making users dizzy.
- Fix: changed to behavior: 'instant' for snappy chapter jumps.

Fix 6 — Send Button Icon (reader.html)
- "Send" text replaced with an up-arrow SVG icon; button resized to 40×40px square
  with flex centering; removed font-size/font-weight/white-space properties.


SESSION 3: PDF Rendering via PDF.js
-------------------------------------
Files: reader3.py, server.py, templates/base.html, templates/reader.html

Root Cause of PDF Issues
- PyMuPDF's get_svg_image() converts text to glyph path outlines (vector shapes),
  not actual SVG <text> elements. Result: fonts rendered wrong (path artifacts),
  text not selectable, not copyable.

Fix — Switch to PDF.js (browser-side rendering)
- reader3.py: process_pdf() now copies source PDF to output_dir/source.pdf via
  shutil.copy2(). Removed all SVG generation. Each chapter's content is a
  <div class="pdf-page-placeholder" data-page="N" style="aspect-ratio:W/H"></div>
  placeholder (aspect-ratio still needed for accurate TOC scroll positions before
  JS renders the canvas).
- server.py: Added GET /read/{book_id}/source.pdf route to serve the PDF file.
  IMPORTANT: route must be defined BEFORE /read/{book_id}/{chapter_index} — FastAPI
  returns 422 (not 404) when {chapter_index: int} fails to parse "source.pdf",
  and does not fall through to the next route.
  Added pdf_needs_reprocess flag: if source.pdf missing (old pickles), template
  shows a re-upload notice instead of broken content.
- templates/base.html: Added {% block head_scripts %}{% endblock %} before </head>
  so child templates can inject <script> tags.
- templates/reader.html: For PDF books:
  - Loads pdf.min.js (PDF.js 3.11.174) from CDN in head_scripts block
  - CSS: .pdf-page-placeholder (sized box before render), .pdf-page-wrapper
    (position:relative container), .textLayer (position:absolute inset:0,
    spans with color:transparent for invisible-but-selectable text)
  - JS: pdfjsLib.getDocument() loads PDF, IntersectionObserver lazily fires
    renderPDFPage() for each chapter-section entering viewport (rootMargin 800px).
    Each page: canvas rendered at container width (scale = containerWidth / pageWidth),
    then text layer rendered via pdfjsLib.renderTextLayer({textContent, container,
    viewport}) for selectable/copyable text. Placeholder replaced with wrapper div.

Result
- Correct font rendering (browser's font engine, not path outlines)
- Text is fully selectable and copyable
- TOC navigation works (aspect-ratio placeholder keeps scroll positions accurate)
- Pages load lazily as user scrolls (800px pre-load margin)


SESSION 4: Text Selection Context for Chat
-------------------------------------------
Files: templates/reader.html, server.py

Feature — Highlight text to use as chat context
- When the user selects/highlights text in the reading area (#main), a quote chip
  appears above the chat input showing a preview of the selected passage (up to 2 lines).
- The chip has a label "Selected:" and an × dismiss button to clear the selection.
- Selecting new text replaces any previous selection.
- On send, the selected text is:
  - Shown as a visual quote block inside the user's chat bubble (italic-styled, with
    a left border) so the conversation history is self-explanatory.
  - Sent to the backend as selected_text field and saved to chat history JSON.
  - Injected into the LLM messages as [Highlighted passage: "..."] prepended to the
    user message, so the model knows exactly which passage is being referenced,
    independent of the chapter text already in the system prompt.
- The quote chip clears automatically on send.
- History reload: loadChatHistory() passes msg.selected_text to appendMessage() so
  previously quoted passages render correctly when reopening a conversation.

Changes:
- reader.html CSS: .selection-quote / .selection-quote-* chip styles,
  .chat-msg.user .quote-block style for the in-bubble quote rendering.
- reader.html HTML: <div id="selectionQuote"> chip inserted above .chat-input-container.
- reader.html JS: selectedText variable + setSelectedText(), mouseup listener on #main,
  dismiss button handler; appendMessage() updated with quotedText 4th param;
  sendMessage() captures quoteContext before clearing, passes it to appendMessage and
  to the fetch payload as selected_text.
- server.py: selected_text extracted from request body; user_content built with
  [Highlighted passage: ...] prefix when present; history entry stores selected_text
  field; history replay also injects the prefix for past messages with selections.
